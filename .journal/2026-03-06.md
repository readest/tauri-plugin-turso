# Journal - 2026-03-06

## 2026-03-06 — Migrate from libsql to turso crate and rename plugin

### Core Decision/Topic
Replaced the deprecated `libsql` crate with `turso` crate (v0.5.0) as the Rust backend, then renamed the entire plugin from `tauri-plugin-libsql` to `tauri-plugin-turso` to reflect the new dependency.

---

### Part 1: libsql → turso crate migration

**Problem**: The `libsql` crate (v0.9.29) is deprecated in favor of the `turso` crate. The API surface changed significantly between the two.

**Key API differences**:

| Concept | libsql (old) | turso (new) |
|---------|-------------|-------------|
| Builder | `libsql::Builder::new_local(path)` | `turso::Builder::new_local(path)` |
| Build | `.build().await?.connect()?` | `.build().await?` → `Database`, then `.connect()?` → `Connection` |
| Execute return | `u32` | `u64` |
| Query | `conn.query(sql, params).await?` → `Rows` | Same, but `Rows` API differs |
| Column names | From `Row` object | From `Rows` object (before iterating) |
| Row values | `row.get::<Value>(i)?` | `row.get_value(i)?` → `Value` |
| Params | `libsql::params![]` macro | `Vec<turso::Value>` (implements `IntoParams`) |
| Encryption | `EncryptionConfig::new(Cipher::Aes256Cbc, Bytes)` | `EncryptionOpts { cipher: String, hexkey: String }` |
| Error | `libsql::Error` | `turso::Error` |

**Gotcha — turso::Params vs IntoParams**: The `turso` crate re-exports a `Params` enum from `lib.rs` AND has `params::Params` in `params.rs`. Only the latter implements `IntoParams` (sealed trait). Solution: use `Vec<Value>` which implements `IntoParams` directly.

**Files changed**:
- `Cargo.toml` — Replaced `libsql = "0.9.29"` with `turso = { version = "0.5.0", default-features = false }`; removed `bytes`, `core`/`replication`/`remote` features
- `src/wrapper.rs` — Complete rewrite using `turso::Builder`, `turso::Connection`, `turso::Value`; removed remote/replica modes and `catch_unwind`; `connect()` now accepts `experimental: &[String]` for Builder feature flags
- `src/models.rs` — `EncryptionConfig` changed to `{ cipher: String, hexkey: String }`; `LoadOptions` gained `experimental: Vec<String>` with `#[serde(default)]`; removed `sync_url`, `auth_token`, `Cipher` enum
- `src/error.rs` — `Error::Libsql(#[from] libsql::Error)` → `Error::Turso(#[from] turso::Error)`
- `src/decode.rs` — Imports changed to `turso::Row`/`turso::Value`; index type `i32` → `usize`
- `src/commands.rs` — Removed `sync` command; `load` now passes `&options.experimental` to `DbConnection::connect()`
- `src/lib.rs` — Removed `commands::sync` from `generate_handler![]`
- `permissions/default.toml` — Removed `"allow-sync"`

### Part 2: Enable FTS (Tantivy-based full-text search)

**Problem**: After migrating to turso, FTS tests were skipping with "unknown module name 'fts'". The `turso` crate doesn't expose `turso_core`'s `fts` cargo feature. Even with `experimental_index_method(true)`, FTS wasn't compiled in.

**Investigation**:
- `turso_core` (v0.5.0) has `fts = ["dep:tantivy"]` feature
- `turso_sdk_kit` depends on `turso_core` but only passes through `encryption`, not `fts`
- `turso` crate only exposes `default = ["mimalloc"]` and `sync` features

**Solution**: Added `turso_core = { version = "0.5.0", default-features = false, features = ["fts"] }` as a direct dependency. Cargo's feature unification merges this with the `turso_core` already pulled in by `turso_sdk_kit`, enabling FTS in the binary.

**Result**: All 51 tests pass — 15 FTS tests that were skipping now run successfully alongside 15 vector search tests and 21 CRUD tests.

### Part 3: Plugin rename (tauri-plugin-libsql → tauri-plugin-turso)

**Scope**: Full rename of directory, Rust crate, IPC plugin name, TypeScript imports, capabilities, and git submodule.

**Changes**:

| Area | Old | New |
|------|-----|-----|
| Directory | `src-tauri/plugins/tauri-plugin-libsql/` | `src-tauri/plugins/tauri-plugin-turso/` |
| Cargo package | `tauri-plugin-libsql` | `tauri-plugin-turso` |
| Plugin name (IPC) | `plugin:libsql\|execute` | `plugin:turso\|execute` |
| Rust struct | `Libsql` | `Turso` |
| Builder registration | `Builder::new("libsql")` | `Builder::new("turso")` |
| Capabilities | `libsql:default` | `turso:default` |
| TS import | `from 'tauri-plugin-libsql'` | `from 'tauri-plugin-turso'` |
| tsconfig path alias | `"tauri-plugin-libsql"` | `"tauri-plugin-turso"` |

**Git submodule rename** required manual steps:
1. Update `.gitmodules` path and section name
2. Move `.git/modules/.../tauri-plugin-libsql` → `.../tauri-plugin-turso`
3. Update `.git` file inside submodule to point to new modules path
4. Update `worktree` path in `.git/modules/.../config`
5. Update git config sections (`--remove-section` old, add new)

**Files outside the plugin that changed**:
- `src-tauri/Cargo.toml` — dependency path
- `src-tauri/src/lib.rs` — `tauri_plugin_turso::init()`

**Test results**: 51/51 Tauri integration tests pass, 15/15 mock unit tests pass.

### Experimental Features Support

Added `LoadOptions.experimental: Vec<String>` to allow enabling turso experimental features at load time. The wrapper maps strings to Builder methods:

```rust
match feature.as_str() {
    "index_method" => builder.experimental_index_method(true),
    "encryption"   => builder.experimental_encryption(true),
    "triggers"     => builder.experimental_triggers(true),
    "attach"       => builder.experimental_attach(true),
    "custom_types" => builder.experimental_custom_types(true),
    "materialized_views" => builder.experimental_materialized_views(true),
    _ => {}
}
```

Frontend usage:
```typescript
await invoke('plugin:turso|load', {
  options: { path: 'sqlite::memory:', experimental: ['index_method'] }
});
```
